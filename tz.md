# ТЗ: мобильный сайт-квиз на день рождения Вани

## 1. Цель проекта

Разработать интерактивный мобильный сайт-квиз для вечеринки. Участники заходят на сайт со своих телефонов, вводят имя, подключаются к комнате квиза и отвечают на вопросы про Ваню.

Квиз запускается организатором. После каждого вопроса игрокам показывается правильный ответ. После завершения всех вопросов отображается финальный рейтинг игроков и статистика распределения ответов.

Главный приоритет UI — корректная и удобная работа на **iPhone 14 Pro, iPhone 16 Pro Max и iPhone 17 Pro Max**.

Для адаптива ориентироваться на следующие CSS viewport-профили:

| Устройство        |                Целевой viewport |
| ----------------- | ------------------------------: |
| iPhone 14 Pro     |         393 × 852 CSS px, DPR 3 |
| iPhone 16 Pro Max |         440 × 956 CSS px, DPR 3 |
| iPhone 17 Pro Max | 440 × 956 logical/CSS px, DPR 3 |

Для iPhone 14 Pro дополнительно тестировать ширину **390 px**, так как разные источники и браузерные режимы могут давать небольшое отличие по высоте/ширине видимой области. ([blisk.io][1])
Для iPhone 16 Pro Max и 17 Pro Max целевой профиль — **440 × 956 CSS/logical px**. ([blisk.io][2])

# 2. Роли пользователей

## 2.1. Организатор

Организатор управляет квизом.

Функции:

* создать комнату квиза;
* увидеть код/ссылку/QR для подключения игроков;
* видеть список подключенных игроков;
* запустить квиз по кнопке;
* вручную перейти к следующему вопросу, если включен ручной режим;
* видеть текущий вопрос, таймер и количество ответивших;
* завершить квиз;
* открыть финальный рейтинг и статистику.

## 2.2. Игрок

Игрок участвует в квизе со своего телефона.

Функции:

* открыть сайт;
* ввести имя;
* подключиться к комнате по коду или ссылке;
* дождаться старта;
* отвечать на вопросы;
* видеть таймер;
* видеть правильный ответ после каждого вопроса;
* видеть свое место в рейтинге после завершения.

# 3. Общий сценарий работы

## 3.1. Создание комнаты

1. Организатор открывает сайт.
2. Выбирает режим **«Я организатор»**.
3. Нажимает **«Создать квиз»**.
4. Система создает комнату и показывает:

    * код комнаты;
    * ссылку для игроков;
    * QR-код;
    * список подключенных игроков;
    * кнопку **«Запустить квиз»**.

## 3.2. Подключение игрока

1. Игрок открывает ссылку или вводит код комнаты.
2. Вводит имя.
3. Попадает на экран ожидания.
4. Видит сообщение в стиле:
   **«Ждем, пока организатор нажмет большую красную кнопку хаоса»**.
5. После запуска квиза автоматически переходит к первому вопросу.

## 3.3. Прохождение вопроса

1. Всем игрокам одновременно показывается вопрос.
2. На ответ дается **30 секунд**.
3. За **10 секунд до конца** должен сработать звуковой сигнал.
4. Игрок выбирает один или несколько вариантов ответа в зависимости от типа вопроса.
5. После выбора ответ фиксируется.
6. Изменить ответ нельзя, если не указано иное в настройках.
7. Если игрок не успел ответить за 30 секунд, ответ считается пропущенным.
8. После завершения таймера всем показывается правильный ответ и комментарий.

## 3.4. Завершение квиза

После последнего вопроса система показывает:

* финальный рейтинг всех игроков;
* количество баллов каждого игрока;
* место каждого игрока;
* количество правильных ответов;
* таблицу распределения ответов по каждому вопросу;
* статистику по вариантам ответа;
* самые популярные неправильные ответы.

# 4. Функциональные требования

## 4.1. Комната квиза

### FR-001. Создание комнаты

Система должна позволять организатору создать новую комнату квиза.

Комната должна иметь:

* уникальный roomId;
* короткий код подключения;
* статус:

    * `WAITING`;
    * `IN_PROGRESS`;
    * `QUESTION_ACTIVE`;
    * `QUESTION_RESULT`;
    * `FINISHED`.

### FR-002. Подключение игроков

Игрок должен подключаться к комнате по ссылке или коду.

При подключении игрок вводит имя.

Ограничения:

* имя от 2 до 20 символов;
* нельзя пустое имя;
* при совпадении имен система добавляет короткий суффикс, например `Иван #2`;
* игрок может переподключиться при обновлении страницы.

### FR-003. Экран ожидания

До запуска квиза игроки должны видеть экран ожидания.

На экране:

* название квиза;
* свое имя;
* статус подключения;
* количество игроков в комнате;
* сообщение, что квиз еще не начался.

# 5. Режим организатора

## 5.1. Экран лобби

Организатор должен видеть:

* код комнаты;
* ссылку для приглашения;
* QR-код;
* список игроков;
* количество подключенных игроков;
* кнопку **«Запустить квиз»**.

## 5.2. Запуск квиза

Квиз должен начинаться только после нажатия организатором кнопки **«Запустить квиз»**.

После запуска:

* комната переходит в статус `IN_PROGRESS`;
* всем игрокам отправляется событие старта;
* начинается первый вопрос.

## 5.3. Управление вопросами

Поддержать два режима:

### Автоматический режим

После показа результата текущего вопроса следующий вопрос начинается автоматически через заданную паузу, например 7 секунд.

### Ручной режим

После показа результата организатор нажимает **«Следующий вопрос»**.

Для MVP можно сделать ручной режим приоритетным, так как на вечеринке организатору удобнее контролировать темп.

---

# 6. Режим игрока

## 6.1. Экран вопроса

Игрок должен видеть:

* номер вопроса;
* текст вопроса;
* варианты ответа;
* таймер;
* индикатор выбранного ответа;
* кнопку подтверждения ответа, если вопрос multiple choice.

Для single choice ответ можно фиксировать сразу после нажатия на вариант.

Для multiple choice игрок выбирает несколько вариантов и нажимает **«Ответить»**.

## 6.2. Таймер

На каждый вопрос дается **30 секунд**.

Таймер должен:

* отображаться визуально;
* синхронизироваться сервером;
* не зависеть от локального времени телефона;
* продолжать корректно работать при кратковременном лаге сети;
* блокировать ответ после истечения времени.

## 6.3. Звуковое уведомление

За **10 секунд до конца вопроса** должен проигрываться звук.

Требования:

* звук должен быть коротким;
* звук не должен ломать UX, если браузер запрещает autoplay;
* после входа игрока желательно запросить первое пользовательское взаимодействие, чтобы звук мог воспроизводиться;
* если звук недоступен, показать визуальное предупреждение:

    * вибрация, если поддерживается;
    * мигание таймера;
    * текст **«10 секунд, думай быстрее, профессор авиации не ждет»**.

## 6.4. Экран результата вопроса

После завершения вопроса игрок видит:

* свой выбранный ответ;
* правильный ответ;
* комментарий к вопросу;
* начисленные баллы;
* текущую позицию в рейтинге, опционально.

Пример комментария:

> Правильный ответ: Ту-160. Огромный, красивый, звучит как диагноз, но это любовь.

# 7. Логика вопросов

Каждый вопрос должен иметь структуру:

```json
{
  "id": "q1",
  "text": "Какой самолет у Вани любимый?",
  "type": "single",
  "options": [
    {
      "id": "A",
      "text": "Су-27"
    },
    {
      "id": "B",
      "text": "Ту-160"
    }
  ],
  "correctOptions": ["B"],
  "difficulty": "medium",
  "points": 2,
  "afterAnswerComment": "Огромный, красивый, как Ванина тяга к матчасти."
}
```

Типы вопросов:

* `single` — один правильный вариант;
* `multiple` — несколько правильных вариантов.

Для multiple choice ответ считается правильным только при полном совпадении множества выбранных вариантов с правильными.

# 8. Подсчет баллов

## 8.1. Базовые правила

* За правильный ответ начисляются баллы вопроса.
* За неправильный ответ — 0 баллов.
* За пропуск — 0 баллов.

## 8.2. Дополнительный бонус за скорость

Можно добавить опционально:

* ответ в первые 10 секунд: +1 балл;
* ответ после 10 секунд: без бонуса.

Для MVP бонус за скорость можно не делать, чтобы не усложнять механику.

## 8.3. Рейтинг

Рейтинг сортируется по:

1. количеству баллов;
2. количеству правильных ответов;
3. среднему времени ответа;
4. времени последнего правильного ответа.

| Место | Игрок | Баллы | Правильных ответов | Среднее время |
| ----: | ----- | ----: | -----------------: | ------------: |

По каждому вопросу показать статистику:

| Вопрос                |  A |  B |  C |  D | Правильный |
| --------------------- | -: | -: | -: | -: | ---------- |
| Любимый самолет Вани? |  2 |  8 |  1 |  0 | B          |

Также желательно показать:

* самый популярный неправильный ответ;
* вопрос, на котором сгорело больше всего людей;
* вопрос, где все ответили правильно;
* вопрос, где никто не понял, что происходит.

# 10. Бесячие UI-моменты

В UI должны быть рандомные раздражающие эффекты, которые иногда мешают читать или отвечать, но не делают игру полностью непроходимой.

## 10.1. Общие правила

Эффекты должны:

* появляться случайно;
* не блокировать интерфейс дольше 1–3 секунд;
* не перекрывать кнопку ответа полностью больше чем на 1 секунду;
* не ломать доступность сайта;
* не вызывать сильное мерцание;
* быть отключаемыми флагом `annoyingModeEnabled`.

## 10.2. Примеры эффектов

### Эффект 1. «Ванина историческая справка»

На экране появляется всплывашка:

> Краткая историческая справка на 30 секунд...

Она частично перекрывает вопрос на 1.5 секунды.

### Эффект 2. «Самолет пролетел»

Через экран пролетает маленький самолет и на секунду закрывает один из вариантов ответа.

### Эффект 3. «МАИ loading...»

Поверх кнопки появляется надпись:

> МАИ думает...

Через 1 секунду исчезает.

### Эффект 4. «ДПП атакует»

Случайная плашка:

> Платформа ДПП хочет оффер мечты

Появляется снизу и слегка двигает layout.

### Эффект 5. «Маша detected»

На экране на 1–2 секунды появляется warning:

> Обнаружена Маша. Ваня напрягся.

### Эффект 6. «РЛЭ Су-27»

Текст вопроса на короткое время заменяется на:

> См. руководство по летной эксплуатации Су-27, глава 18.

Потом возвращается обратно.

## 10.3. Ограничения для бесячих эффектов

Не допускается:

* полностью скрывать все варианты ответа;
* мешать нажать ответ в последние 5 секунд;
* воспроизводить громкие звуки без действия пользователя;
* делать частые вспышки;
* использовать эффекты, которые могут вызвать дискомфорт у чувствительных пользователей.

# 13. Realtime-логика

## 13.1. Основной принцип

Сервер является источником истины.

Сервер отвечает за:

* статус комнаты;
* текущий вопрос;
* время начала вопроса;
* время окончания вопроса;
* прием ответов;
* блокировку ответов после таймера;
* расчет результатов;
* рассылку событий игрокам.

Клиент не должен сам решать, истекло ли время. Клиент может только отображать оставшееся время на основе серверного `questionEndsAt`.

## 13.2. WebSocket-события

### События от клиента

```text
JOIN_ROOM
CREATE_ROOM
START_QUIZ
SUBMIT_ANSWER
NEXT_QUESTION
FINISH_QUIZ
PING
```

### События от сервера

```text
ROOM_CREATED
PLAYER_JOINED
PLAYER_LEFT
QUIZ_STARTED
QUESTION_STARTED
ANSWER_ACCEPTED
ANSWER_REJECTED
QUESTION_FINISHED
QUESTION_RESULT
QUIZ_FINISHED
ROOM_STATE_SYNC
ERROR
```

## 13.3. Пример события начала вопроса

```json
{
  "type": "QUESTION_STARTED",
  "payload": {
    "roomId": "ABCD",
    "questionId": "q12",
    "questionNumber": 12,
    "totalQuestions": 40,
    "text": "Что Ваня выберет: редкий самолет или оффер мечты?",
    "options": [
      {
        "id": "A",
        "text": "Редкий самолет"
      },
      {
        "id": "B",
        "text": "Оффер мечты"
      }
    ],
    "type": "single",
    "startedAt": "2026-05-07T18:00:00Z",
    "endsAt": "2026-05-07T18:00:30Z"
  }
}
```

# 16. Стек технологий

Рекомендуемый стек:

* TypeScript;
* React;
* Vite;
* Tailwind CSS;
* Hero UI v3

Рекомендуемый стек:

* Kotlin 2.2.x;
* Spring Boot 4.x;
* Spring WebSocket;
* Spring WebFlux;
* Spring Data R2DBC;
* PostgreSQL;
* Redis для live-state комнат и TTL;
* Flyway для миграций;
* Gradle Kotlin DSL;
* Docker Compose для локального запуска.


## 16.3. Инфраструктура

Для MVP:

* frontend и backend можно поднять через Docker Compose;
* PostgreSQL — для вопросов, игроков, результатов;
* Redis — для состояния активной комнаты;
* Nginx/Caddy — reverse proxy;
* HTTPS обязателен, иначе на iOS могут быть проблемы с некоторыми browser API.

# 17. Нефункциональные требования

## 17.1. Производительность

Система должна поддерживать минимум:

* 1 активную комнату;
* 30 игроков одновременно;
* 40 вопросов;
* отправку realtime-событий без заметной задержки.

Целевая задержка WebSocket-событий:

```text
до 300 мс внутри нормальной сети
```

## 17.2. Надежность

Система должна:

* переживать обновление страницы игроком;
* позволять игроку переподключиться;
* не терять уже отправленный ответ;
* не принимать два ответа от одного игрока на один вопрос;
* не принимать ответ после истечения времени;
* корректно завершать вопрос, даже если часть игроков не ответила.

## 17.3. Безопасность

Нужно предусмотреть:

* защиту комнаты коротким кодом;
* отдельный organizerToken;
* невозможность игроку отправить команду старта квиза;
* валидацию всех WebSocket-команд на backend;
* rate limit на подключение и отправку событий;
* очистку HTML в именах игроков.

# 18. Acceptance Criteria

## AC-001. Организатор может создать комнату

Дано: организатор открыл сайт.
Когда: он нажимает **«Создать квиз»**.
Тогда: система создает комнату, показывает код, ссылку и QR.

## AC-002. Игрок может подключиться

Дано: комната создана.
Когда: игрок вводит имя и код комнаты.
Тогда: он попадает в лобби ожидания.

## AC-003. Квиз стартует только по кнопке организатора

Дано: игроки подключены.
Когда: организатор нажимает **«Запустить квиз»**.
Тогда: всем игрокам показывается первый вопрос.

## AC-004. На вопрос дается 30 секунд

Дано: вопрос начался.
Когда: проходит 30 секунд.
Тогда: прием ответов блокируется, вопрос завершается.

## AC-005. За 10 секунд до конца есть уведомление

Дано: до конца вопроса осталось 10 секунд.
Тогда: проигрывается звук или показывается визуальное предупреждение.

## AC-006. После вопроса показывается правильный ответ

Дано: вопрос завершен.
Тогда: игрок видит правильный ответ, свой ответ и комментарий.

## AC-007. После квиза показывается рейтинг

Дано: завершен последний вопрос.
Тогда: система показывает рейтинг всех игроков.

## AC-008. После квиза показывается распределение ответов

Дано: квиз завершен.
Тогда: система показывает таблицу распределения ответов по вопросам и вариантам.

## AC-009. Бесячие эффекты появляются случайно

Дано: игрок отвечает на вопрос.
Когда: включен annoying mode.
Тогда: случайные UI-эффекты могут появляться, но не должны полностью блокировать возможность ответа.

## AC-010. Сайт корректно работает на iPhone

Дано: пользователь открыл сайт на iPhone 14 Pro, 16 Pro Max или 17 Pro Max.
Тогда: весь основной игровой сценарий проходит без горизонтального скролла, перекрытия кнопок системными областями и проблем с таймером.
